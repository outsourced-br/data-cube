Outsourced.DataCube

One analytical engine inside your application

Outsourced.DataCube is an in-memory analytics cube for .NET. It helps teams model business data once through dimensions, metrics, and fact groups, then reuse that model for slicing, grouping, aggregating, and pivoting data across many use cases.

Why use it?

Most applications do not struggle to store data. They struggle to turn the same data into many consistent analytical views. Teams end up rewriting similar calculations in SQL, LINQ, services, exports, dashboards, and scripts.

The result is drift: revenue means one thing here, another thing there, and percentages get calculated incorrectly because the logic is spread everywhere.

Use the Data Cube when the same business data is analyzed from many angles and you want one reusable, testable, and consistent way to turn that data into decisions.

In practical terms

  • Define the analytical structure once
  • Reuse dimensions and metrics across modules
  • Reduce ad hoc grouping logic
  • Improve trust in numbers
  • Ship new analyses faster

What real-life problems does it solve?

Inconsistent KPIs

The same metric often gets implemented differently across reports and services. A cube creates one analytical model instead of many local versions.

Slow report delivery

Once the cube exists, new analytical views often become new slices over the same model instead of new one-off pipelines.

Hard root-cause analysis

Business questions rarely stop at totals. People want to move from overall numbers to category, week, product, hour, or tag. That is what a cube is built for.

Fragile analytical code

Ad hoc GroupBy-heavy code becomes repetitive and hard to maintain. A cube gives that logic a cleaner home.

How to explain it to non-technical people

The pivot table analogy

It is like having a reusable pivot-table engine inside the product instead of rebuilding pivot logic again and again.

The contract analogy

It creates a shared contract for what the dimensions are, what the metrics are, and how those metrics are allowed to aggregate.

The foundation analogy

Instead of building every report from raw bricks, you build a stable analytical foundation that many reports can stand on.

Who should use it?

Developers

Use it when your application needs reusable business analysis and you want to stop scattering analytical logic across the codebase.

Architects and technical leads

Use it when you want a shared analytical model that survives beyond one report, one endpoint, or one sprint.

Product and solution people

Use it when the business keeps asking the same data to be viewed in different ways and you need a model, not another custom export.

Typical use cases

  • sales by product, week, hour, or channel
  • pricing and margin analysis
  • inventory and stock-risk analysis
  • category and taxonomy slicing
  • trend detection and time-window comparisons
  • embedded operational dashboards in business applications

What it is not

It is not a hosted BI platform, not a server-side query language, and not an enterprise OLAP server. It is an in-memory analytical engine for .NET applications.

That is deliberate. It stays close to the application, easy to test, and easy to evolve.

Quick example

var cube = new AnalyticsCube
{
  Key = "sales",
  Label = "Sales Analysis"
};

var product = cube.AddTypedDimension<string>("product", "Product");
var week = cube.AddTypedDimension<string>("week", "Week");
var revenue = cube.AddCurrencyMetric("revenue", "BRL", "Revenue");

cube.CreateFactGroup()
  .WithDimensionValue(product, "Cola")
  .WithDimensionValue(week, "2026-W14")
  .WithMetricValue(revenue, 1250m)
  .Build();
